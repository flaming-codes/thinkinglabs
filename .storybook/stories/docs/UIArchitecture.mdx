# UI architecture and Storybook boundaries

The shared UI package now lives in `src/frontend/thinkinglabs-ui/` and is the source of truth for Storybook-rendered UI.

## Package map

| Path                                       | Owns                                 | Notes                                                                                 |
| ------------------------------------------ | ------------------------------------ | ------------------------------------------------------------------------------------- |
| `src/frontend/thinkinglabs-ui/components/` | Reusable primitives and shell pieces | `StatusTag`, `ConfidenceMeter`, `MetricTile`, `SiteHeader`, `AppShell`, chart helpers |
| `src/frontend/thinkinglabs-ui/pages/`      | Page-level compositions              | Compose primitives into full page surfaces for Storybook and eventual frontend reuse  |
| `src/frontend/thinkinglabs-ui/types.ts`    | UI-facing contracts                  | Shared prop/data shapes (`ClaimSummary`, `PostDetail`, `NowData`, etc.)               |
| `src/frontend/thinkinglabs-ui/styles.css`  | Design tokens + responsive rules     | `:root` fluid type, color, and spacing tokens plus breakpoint layout changes          |
| `src/frontend/thinkinglabs-ui/mocks/`      | Storybook fixture data               | Re-exported through `mocks/index.ts`; use in stories only                             |
| `.storybook/stories/*.stories.ts`          | Story definitions                    | Import shared UI from `../../src/frontend/thinkinglabs-ui/...`                        |

## Module boundaries (DRY guardrails)

- **Components stay presentation-only**: no content collection reads inside `components/`.
- **Compositions own layout + composition**: `pages/*Composition.astro` wires primitives together with typed props.
- **Frontend routes adapt real data**: map content data in `src/pages/**` and pass typed props into compositions.
- **Mocks are not runtime data**: `src/frontend/thinkinglabs-ui/mocks/*` is for Storybook fixtures.

## Import conventions after relocation

Storybook stories should import from the relocated UI package:

```ts
import ClaimsListingPageComposition from "../../src/frontend/thinkinglabs-ui/pages/ClaimsListingPageComposition.astro";
import { CLAIMS } from "../../src/frontend/thinkinglabs-ui/mocks";
```

Real frontend routes should prefer the `@/` alias:

```astro
---
import ClaimsListingPageComposition from "@/frontend/thinkinglabs-ui/pages/ClaimsListingPageComposition.astro";
import type { ClaimSummary } from "@/frontend/thinkinglabs-ui/types";
---
```

## Integration pattern for real frontend pages

1. Read/derive data in `src/pages/**` (or `src/lib/**`) from `astro:content`.
2. Map the data into `src/frontend/thinkinglabs-ui/types.ts` contracts.
3. Render the composition with typed props.
4. Keep data fetching and transformation outside shared UI components.

Example route adapter shape:

```astro
---
import ClaimsListingPageComposition from "@/frontend/thinkinglabs-ui/pages/ClaimsListingPageComposition.astro";
import type { ClaimSummary } from "@/frontend/thinkinglabs-ui/types";

const claims: ClaimSummary[] = /* map getCollection("claims") -> UI contract */;
---

<ClaimsListingPageComposition claims={claims} totalCount={claims.length} />
```

When adding a new page:

1. Add/extend types in `types.ts`.
2. Build/update a composition in `pages/`.
3. Add fixture data in `mocks/` (if Storybook needs isolated examples).
4. Add/update `.storybook/stories/*.stories.ts`.
5. Integrate in real `src/pages/**` by adapting real data to the same contract.
