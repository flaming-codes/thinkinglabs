# Rendering pipeline

Listing pages, detail pages, the per-collection JSON APIs under `/api/`, contract-validated `.md` page variants, and `public/llms.txt` all derive from Astro's typed `getCollection(kind)` over `content/<kind>/` (validated by `src/schemas/`) and the kind registry in `src/lib/registry.ts`. Static Markdown summaries are maintained explicitly in `src/lib/markdown-routes.ts`; broader public-surface discovery for nav and `llms.txt` still lives in `src/lib/surfaces.ts`. Agent-facing derived metadata (`summary`, `word_count`, `approx_token_count`, source path, Markdown URL, and GitHub source URL) lives in `src/lib/agent-metadata.ts` and is projected into JSON APIs, Markdown envelopes, the SQLite index, and MCP responses. A new public surface is added by appending an entry to `surfaces.ts`; nav and `llms.txt` pick it up automatically on the next build. A new object kind requires all of: a Zod schema in `src/schemas/<kind>.ts`; registration in `src/schemas/index.ts` (`KIND_SCHEMAS`); a `KIND_REGISTRY` entry in `src/lib/registry.ts` (this is the keystone - it governs the route, title/date fields, nav inclusion, API exposure, and MCP resource registration; omitting it leaves the kind with no route, no nav entry, and no MCP view); declaring the collection in `src/content.config.ts`; adding listing + detail pages; and a one-line `src/pages/api/<kind>.json.ts` that re-exports `collectionJson("<kind>")`. Compile-time coverage assertions in `src/content.config.ts` and `src/lib/registry.ts` catch missing entries in `KIND_SCHEMAS` and `KIND_REGISTRY`.

The site renderer never reads `dist/index.sqlite`; the index is for agents. Schemas are the single source of typing for both runtime validation and `getCollection` return types. Per-kind logic stays in the page file (or a kind-specific helper in `src/lib/`). Shared visual components live in `src/frontend/thinkinglabs-ui/components/` (e.g. `StatusTag.astro`, `EntityDetail.astro`, `EntityFacts.astro`); `src/components/` is the infrastructure/utility layer (`JsonLd.astro`, `PwaHead.astro`).

Entity detail heroes are local assets resolved by convention, not frontmatter. To give an entity a custom hero, add `src/assets/<kind>/<slug>.<ext>` where `<kind>` is the route folder and `<slug>` is the content file slug. Supported extensions are `avif`, `webp`, `png`, `jpg`, and `jpeg`, chosen in that order when multiple files exist. Missing custom assets fall back to `src/assets/hero.png`. The same convention feeds the rendered detail page and the generated OG image.

Markdown bodies pass through two custom plugins wired in `astro.config.mjs`. `src/markdown/remark-section-freshness.ts` lifts Pandoc-style heading attributes (`## Title {#id last_verified="YYYY-MM-DD"}`) into `hProperties`. `src/markdown/rehype-section-freshness.ts` appends a `<span class="freshness-pill">` to any heading carrying `data-last-verified`, color-coded by age via `src/lib/freshness.ts`. Both plugins no-op on content that lacks the syntax, so they cost effectively nothing on non-posts kinds. The build-time "now" honors `FRESHNESS_NOW_ISO` for deterministic tests and local builds.

## Markdown URL variants

`src/pages/[...slug].md.ts` prerenders Markdown siblings for canonical public page and content routes. The route inventory and serializers live in `src/lib/markdown-routes.ts`: selected static pages are listed explicitly, listing pages come from public listing kinds, and detail pages come from each public collection. Non-HTML and data surfaces stay outside this route family, including `/api/*.json`, `/og/*.png`, `/sitemap.xml`, `/robots.txt`, `/healthz`, `/feed/*`, `/llms.txt`, and `/contact.json`.

The `.md` output has its own small Zod contracts for detail, listing, and static-page envelopes. Those contracts validate the public YAML envelope (`variant`, route metadata, title/description, kind/slug/count as appropriate, and resolved or explicitly unresolved links for detail pages) without duplicating the per-kind source schemas. Detail pages preserve the canonical `entry.body` after the envelope; listing pages synthesize compact Markdown from registry metadata; static pages are maintained summaries.

Detail Markdown envelopes include `agent_metadata`, including `approx_token_count`. The estimate is deliberately rough (`chars/4`) and model-independent; it exists to help agents budget context, not to promise tokenizer-specific precision. Detail pages also expose a Markdown alternate link in `<head>` and an `EntityDetail`-level "Copy for AI" control that copies the clean `.md` sibling instead of rendered HTML.

## Claim history rendering

`/claims/<slug>` renders a confidence-over-time history table by calling `walkFileHistory(cwd, repoRelativePath)` from `src/lib/git.ts` and parsing each revision via `parseClaimHistory` in `src/lib/claim-history.ts`. The history is read-only at render time; all writes to claim files happen through the derivation CLI (`scripts/derive-claims.ts`) or the stale-review CLI (`scripts/review-stale-claims.ts`), neither of which is invoked at build time.
