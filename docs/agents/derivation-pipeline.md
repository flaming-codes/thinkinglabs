# Derivation pipeline

Thoughts under `content/thoughts/` are the input; claims under `content/claims/` are the output. The link is bidirectional: a thought's `claims:` frontmatter array lists the slugs of claims derived from it; each claim's `derived_from` array points back to `thoughts/<slug>`. Both directions are traversable at build time and by agents querying the index. `scripts/derive-claims.ts` is the orchestrator: it collects thought paths (filtered by `--thought <slug>`, `--all`, or explicit file path), loads existing claims for merge detection, calls `proposeClaimsForThought` in `src/lib/derive-claims.ts` (`tier: "balanced"` via `src/lib/llm.ts`, forced tool-use structured output via `extract_claim_proposals`, Zod-validated at the boundary), then hands the proposal list to `runReview` (`src/lib/review-cli.ts`) with five action handlers. Accept and edit each call `proposalToClaimFile` — a manual YAML emitter, narrow on purpose for deterministic key ordering (see ADR-007) — then write the claim file, call `addClaimBacklink` (`src/lib/backlinks.ts`) to patch the source thought, and write an accepted AI provenance event to `content/provenance/` using the `ModelRef` captured at proposal time. Reject stores a SHA-256 hash of `(thoughtSlug, claimText)` in `.derivation-rejections.json`; defer stores the full proposal in `.derivation-deferrals.json`; both files are gitignored. Merge augments an existing claim's evidence/opposing via gray-matter round-trip rather than creating a new file and writes provenance only when the merge is accepted.

Where to look: the LLM system prompt lives in `src/lib/derive-claims.ts` alongside the Zod tool-output schema; file emission is `proposalToClaimFile` in the same file; state bookkeeping (`DerivationState`, rejection hashes, deferrals) is in `scripts/derive-claims.ts`; the keystroke runtime is `src/lib/review-cli.ts`; backlink writes are `src/lib/backlinks.ts`. The sibling pipeline, `scripts/review-stale-claims.ts` (Slice C), reuses `runReview` and `editInEditor` for stale-claim re-review; consult it alongside this pipeline for the full picture of how claims move through their lifecycle.

M5 agents (`dormant-flip`, `review-decisions`, `resolve-predictions`, `freshness-review`, `triage-questions`) follow the proposal/confirmation pattern documented in ADR-009; they emit to the queue rather than mutating directly.
