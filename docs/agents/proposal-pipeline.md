# Proposal pipeline

The proposal queue lives at `.proposal-queue.json` in the repo root (gitignored). Background agents scan content collections and call `enqueue` from `src/lib/proposal-queue.ts` to append typed `QueuedProposal` objects; they never write to `content/` directly. `proposalId(source, type, target, payload)` produces a deterministic SHA-256 key, so re-running an agent over unchanged content produces the same ids and deduplicates naturally. Agent modules live at `src/lib/agents/<agent>.ts`; each registers a `ProposalHandler` at module-import time by calling `registerHandler` from `src/lib/proposal-dispatch.ts`. The handler declares a `payloadSchema` (Zod), a `parse` method, an `apply` method (accepts the proposal — mutates the content tree), an `edit` method (opens the file in `$EDITOR` for human revision before writing), and an optional `reject` hook (records rejection memory in `.<agent>-rejections.json` so the proposal is not re-enqueued on the next scan unless the underlying content changes).

To add a new agent: create `src/lib/agents/<agent>.ts` with a `run<Agent>` export and a `registerHandler` call at the bottom; create `scripts/<agent>.ts` as the CLI entry point with `--no-llm` and `--cwd` flags; add `"<agent>": "tsx scripts/<agent>.ts"` to the `scripts` block in `package.json`; add `import "../src/lib/agents/<agent>.ts"` to `scripts/review-proposals.ts` so the handler is registered when the review CLI runs; and optionally add a launchd plist under `scripts/launchd/` following the skeleton of the existing five plists. The new agent's proposals will appear in `pnpm review-proposals` automatically once the import is in place.

For the _why_ behind this pattern — the scheduling choice, the dedup contract, the `freshness-review` rejection no-op, and the split between agent (schedulable) and `review-proposals` (human-only) — see ADR-009. This document covers the _how_.
